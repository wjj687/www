# 《诗经》全书解释补全设计

## 目标

将全书 `305` 首诗的“解释”功能补齐到与《关雎》《葛覃》同等级的可读体验，并在不破坏现有页面结构的前提下，把解释对象从原先偏窄的“人物 / 地名 / 动植物”扩展为更适合《诗经》全书的百科式解释系统。

完成后，每首诗都应满足以下最低标准：

- 至少有一组可点击的解释实体
- 左侧原文中存在可见、高亮、可点击的标注词
- 右侧解释面板能展示对应说明
- 动植物优先展示国内可访问图片
- 非动植物条目提供清楚、可读、接近百科风格的说明

## 范围

本次补全覆盖：

- 全部 `305` 首诗的解释页数据
- 解释页的数据结构扩展
- 解释页分类显示与交互兼容
- 解释页对新增类别的样式支持

本次不包含：

- 重新设计解释页整体布局
- 为每个非动植物实体强制补图
- 单独引入外部数据库或 CMS

## 设计选择

### 方案比较

1. 扩展解释类别并沿用现有解释页交互  
   这是推荐方案。它能保留现在的左右双栏阅读体验，同时让器物、礼制、制度、天象等词条也能自然进入解释系统。

2. 保留原四类，强行把新增对象塞进旧类别  
   改动更小，但语义会越来越乱，后续维护成本高。

3. 只补右侧解释文案，不做左侧精确标注  
   速度快，但无法达到“按第一二首标准补全”的目标。

### 采用方案

采用方案 1：

- 扩展解释类别
- 保留现有解释页结构
- 让全书解释数据按统一格式进入系统

## 数据模型

### 解释类别

现有类别：

- `person`
- `place`
- `flora`
- `fauna`

新增类别：

- `artifact`：器物、服饰、车马、乐器、采集工具等
- `rite`：礼制、祭祀、婚礼、聘问、朝会等
- `institution`：制度、官职、政治结构、社会秩序
- `celestial`：星宿、天象、历法节令

### 单条解释实体结构

继续沿用现有 `PoemInterpretationEntity` 基础结构：

- `id`
- `label`
- `category`
- `summary`
- `facts`
- `details`
- `image`

约束如下：

- `summary` 用一句到两句交代该词条是什么
- `facts` 用于快速列出识别信息，如“性质 / 所属 / 常见解释 / 时代背景”
- `details` 至少两段，说明语源、语境和在诗中的作用
- `image` 只在确有必要且资源稳定时填写

### 原文标注结构

继续沿用现有：

- `interpretationMentions`

规则如下：

- 优先标特指词或可稳定定位的词组
- 同一行允许多个实体
- 不强求每句都标，但要保证整首诗解释页不为空

## 页面与交互

### 左侧原文

- 保持现有逐行阅读结构
- 保持点击高亮词后，右侧切换对应解释
- 新增类别后仍按颜色区分
- 不增加浮层，不打断阅读

### 右侧解释面板

- 保持现有“标题 / 分类 / 图片 / 摘要 / facts / details”结构
- 分类标签文案改为适配八类解释对象
- 无图片条目直接显示文本说明，不留空白图片区

### 分类显示

解释页图例从 4 类扩展为 8 类，但表现要克制：

- 颜色区分清楚
- 亮度与饱和度控制在不伤阅读的范围
- 不让分类色压过正文墨色

## 内容标准

### 解释对象的选择规则

优先级如下：

1. 特指人名、国名、地名、山川、河流、邦国
2. 明确的动植物
3. 关键器物、服饰、乐器、车马
4. 关键礼制与祭祀语汇
5. 关键制度和官职
6. 关键天象、节令、星宿

如果某首诗缺少明显人物或地名，也必须至少从后四类中选择可解释对象，避免解释页为空。

### 文本风格

- 使用接近百科条目的说明性文字
- 优先解释“这是什么”
- 再解释“它在诗里为什么重要”
- 避免空泛抒情和过度象征化

### 图片策略

动植物优先配图，来源要求：

- 优先选国内可访问来源
- 只使用当前页面可稳定加载的直链
- 同一对象允许复用既有稳定图片资源

非动植物一般不强制配图，除非：

- 有稳定可访问的文物、地理遗址、器物图
- 且图片确实有助于理解

## 实施策略

### 第一阶段：扩展基础结构

- 扩展 `InterpretationCategory`
- 更新解释页图例与分类样式
- 确保新增类别在解释页上能正常显示

### 第二阶段：批量补齐全书数据

- 以现有前两首和已补样例为模板
- 按批次为每首诗补 `interpretationEntities`
- 同步补 `interpretationMentions`
- 动植物优先补图

### 第三阶段：质量校对

- 检查解释页是否为空
- 检查标注词能否正确点击
- 检查图片链接是否有效
- 检查是否有明显错类或空说明

## 风险与处理

### 风险 1：并非每首诗都适合做大量实体标注

处理：

- 不追求每首诗标很多点
- 但必须保证每首诗至少有一组高质量解释对象

### 风险 2：图片来源不稳定

处理：

- 只对高价值动植物条目补图
- 宁可无图，也不接不稳定图

### 风险 3：类别扩展后样式混乱

处理：

- 复用现有解释页视觉语言
- 新增类别只做轻量颜色扩展，不改页面骨架

## 验收标准

- `305` 首诗全部具备解释页数据
- 每首诗解释页均有可点击标注
- 新增类别能正确显示和切换
- 现有解释页交互不回退
- 动植物图片在主要条目上可正常显示
- `npx tsc --noEmit` 通过
- `npm run build` 通过
