# 诗经项目移动化设计

## 目标

把当前 `Next.js` 项目逐步变成手机上可安装、可稳定使用的软件，同时尽量复用现有代码与界面。

第一阶段目标：

- 在手机浏览器中具备接近 App 的体验
- 支持添加到主屏幕
- 支持基础离线访问
- 主要阅读界面在手机竖屏下可正常使用

第二阶段目标：

- 通过 `Capacitor` 封装为 Android App
- 后续具备扩展到 iOS 的可能

## 现状

当前项目是一个基于 `Next.js 15 + React 19` 的阅读型站点，已经具备完整的诗歌、注释、翻译、解释、赏析、AI 解析等页面与交互。项目主要价值集中在现有前端与数据层，因此移动化方案应优先复用现有 Web 代码，而不是重写为原生应用。

## 方案选择

### 方案 A：仅做 PWA

优点：

- 改动最小
- 上线最快
- 维护成本低

缺点：

- 原生能力有限
- 应用商店发布能力较弱

### 方案 B：PWA + Capacitor

优点：

- 最大限度复用现有代码
- 可以打包成真正的 Android / iOS 应用
- 后续可以按需接入原生能力

缺点：

- 需要额外维护移动端壳工程

### 方案 C：重写为 React Native / Flutter

优点：

- 原生能力最强

缺点：

- 成本最高
- 基本等于重做

## 推荐方案

采用 **方案 B：PWA + Capacitor**。

先把当前站点做成完整可安装的移动端 Web App，再生成 Android 应用壳。这样既能尽快得到手机可用的软件版本，也为后续上架和原生增强留出空间。

## 实施范围

### 1. PWA 基础能力

- 增加 Web App Manifest
- 配置应用名称、主题色、图标、启动方式
- 增加 Service Worker 或等效缓存方案
- 提供基础离线访问能力

### 2. 手机端界面适配

- 优化首页、诗歌页在窄屏下的版心与留白
- 调整目录抽屉在手机上的打开、滚动和关闭体验
- 调整右上角功能菜单在小屏上的尺寸与展开方式
- 优化解释、赏析、AI 解析等独立页面的阅读宽度与返回按钮位置

### 3. Capacitor 封装准备

- 统一站点运行入口
- 确认静态资源与路由适配策略
- 生成 Android 工程
- 验证本地打包与运行

### 4. 后续增强

- 增加启动页
- 增加本地收藏持久化优化
- 提升离线缓存策略
- 根据需要接入分享、文件导出等原生能力

## 关键设计决策

### 保持 App Router 结构

现有项目已经围绕 `app/` 目录组织，移动化阶段不改变整体路由结构，只做适配和增强，避免引入大规模重构。

### 优先保证阅读体验

这个项目的核心是“读”。因此移动化优化应优先保证以下页面：

- 诗歌正文页
- 解释页
- 赏析页
- AI 解析页

功能按钮、目录、收藏等属于辅助交互，排在第二优先级。

### 先 Android，后 iOS

Android 更容易在现有环境中验证与分发，因此先实现 Android 封装。iOS 留待具备 macOS / Xcode 环境后再推进。

## 风险与处理

### 风险 1：Next.js 与移动壳整合复杂

处理：

- 先完成 PWA
- 再用最小方式接入 `Capacitor`
- 保持改动集中在配置层与外壳层

### 风险 2：小屏页面密度过高

处理：

- 分页检查重点页面
- 优先压缩边距、优化按钮尺寸、控制文本框宽度

### 风险 3：离线缓存与大体量数据冲突

处理：

- 第一阶段只缓存关键页面和基础静态资源
- 大体量 PDF 等资料不进入首轮离线缓存

## 验收标准

- 手机浏览器可正常访问，界面不破版
- 可添加到主屏幕
- 主页面与核心阅读页面具备 App 感
- Android 外壳工程可以成功生成
- 至少能在本地设备或模拟器中启动运行

## 实施顺序

1. 增加 PWA 配置与图标资源
2. 完成手机端 UI 适配
3. 接入 `Capacitor`
4. 生成并验证 Android App
5. 视情况推进 iOS
