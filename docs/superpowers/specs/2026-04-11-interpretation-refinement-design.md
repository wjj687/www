# 《诗经》解释系统精修设计
## 目标

在保留现有解释页双栏阅读体验的前提下，重做全书 `305` 首诗的解释标准，使解释系统收缩为更明确的四类对象：

- `人物`
- `地名`
- `植物`
- `动物`

这次调整不再追求“凡可解释者皆入解释页”，而是转向“只保留真正适合做解释的实体，并把内容写得更深、更稳、更像通顺的百科说明文”。完成后应满足以下目标：

- 解释页只展示 `人物 / 地名 / 植物 / 动物`
- 原先的 `器物 / 礼制 / 制度 / 天象` 全部移除
- 如果某首诗没有上述四类解释对象，解释按钮仍保留，但进入页面后显示干净的空态说明
- 右侧解释内容更侧重“事物本身是什么”，而不是它在诗里的象征意义或修辞作用
- 保留现有的点击高亮、左右联动、动植物图片能力

## 范围

本次改造包含：

- 全书解释数据的类别裁剪
- 旧解释数据的运行时过滤
- 解释生成脚本的输出标准调整
- 解释页空态行为补齐
- 现有 `人物 / 地名 / 植物 / 动物` 说明文的整体增厚与重写标准

本次改造不包含：

- 重做解释页整体布局
- 把解释页改成百科网站式分栏目录结构
- 为无解释诗篇新增不必要的解释对象
- 重新引入 `器物 / 礼制 / 制度 / 天象`

## 设计选择

### 方案比较

1. 只做类别过滤，不重写内容  
   改动最小，但剩下的解释文案仍然偏薄，达不到“更详尽”的要求。

2. 保留现有界面，重做解释数据标准  
   这是推荐方案。它不推翻已有交互，但会把类别、空态、文本风格和生成逻辑一起收紧，最符合当前目标。

3. 连界面结构也一起改成百科词条页  
   可读性未必更好，且会让任务范围膨胀到整页重构，不适合当前阶段。

### 采用方案

采用方案 2：

- 保留现有解释页双栏结构
- 只保留四类解释对象
- 没有四类解释对象的诗篇显示空态页
- 统一把解释内容改写为更详尽的说明文

## 数据模型

### 保留的解释类别

解释系统只保留以下类别：

- `person`
- `place`
- `flora`
- `fauna`

以下类别从生成层和展示层一并移除：

- `artifact`
- `rite`
- `institution`
- `celestial`

### 单条解释实体结构

继续沿用现有 `PoemInterpretationEntity` 的基本结构：

- `id`
- `label`
- `category`
- `summary`
- `facts`
- `details`
- `image`

但字段内容标准调整为：

- `summary`  
  用 1 到 2 句直接说明“这是什么”。优先交代所指对象、别称、基本身份，不写空泛判断。

- `facts`  
  仅保留高信息密度条目，如别名、分布、物种属性、历史归属、现实地理归属等。避免堆空洞标签。

- `details`  
  作为主体说明文，改为 2 到 4 段连续文字。重点解释对象本身，不再以“诗中作用”作为主轴。必要时最多在结尾轻轻补一句与本篇的联系。

- `image`  
  仅在 `flora / fauna` 默认使用。`person / place` 不强制配图；若后续已有稳定资源可再补。

### 标注结构

继续使用现有 `interpretationMentions`。

调整后的标注规则：

- 只标 `人物 / 地名 / 植物 / 动物`
- 优先标可稳定定位的词或短语
- 不为了凑数量强行拆词
- 若本篇没有这四类对象，允许完全没有高亮标注

## 内容标准

### 解释对象的选取原则

只保留真正适合这四类的对象：

1. 明确的人名、氏族、历史人物、传说人物
2. 明确可指认的国名、地名、山川、河流、地域称呼
3. 明确的植物
4. 明确的动物

以下内容不再纳入解释对象，即使旧数据里曾经出现过：

- 器物
- 礼制
- 制度
- 官职
- 星象
- 抽象概念

### 文本风格标准

解释文字统一改成通顺、详尽的说明文，风格接近百科式介绍，但不堆砌分节标题。

每条解释建议按这个顺序自然展开：

1. 先说明“它是什么”
2. 再说明别称、来源、现实身份或历史背景
3. 再说明外形、习性、用途、分布、流变等核心信息
4. 如果确有必要，再补一句它为何会出现在这首诗里

明确避免以下写法：

- 为了显得“有文采”而空泛抒情
- 过度强调象征意味
- 大量使用“在诗中象征着……”作为主体
- 只写一两句泛泛介绍
- 为了凑结构而加入意义不大的段落

### 动植物说明标准

动植物解释要优先解释对象本身：

- 物种身份或常见归属
- 主要外形特征
- 生活环境或分布
- 在古代文献中的常见称呼或异名
- 与现代常见认知之间的差别

不把“诗中寓意”写成主体，只在必要时略提。

### 人物与地名说明标准

人物说明优先交代：

- 其具体身份
- 所处时代或传说背景
- 史籍中的基本信息
- 名称在不同文献中的说法

地名说明优先交代：

- 对应的真实地理范围或历史区域
- 名称来源或沿革
- 与今天地理认知之间的关系
- 文献中常见的定位方式

## 页面行为

### 有解释对象时

- 左栏保留高亮词与点击切换
- 右栏显示对应解释卡片
- 只出现四类颜色与标签

### 无解释对象时

解释按钮保留。

进入解释页后：

- 左栏不显示高亮原文交互
- 右栏显示统一空态说明卡片
- 文案固定为：`本篇暂无人物、地名或动植物解释。`

这样用户路径保持一致，也不会因为某篇无解释对象而出现断层。

## 实施策略

### 第一阶段：类别裁剪

- 在生成层移除 `artifact / rite / institution / celestial`
- 在合并层过滤旧数据中的上述类别
- 在展示层只保留四类标签与颜色

### 第二阶段：空态兜底

- 为解释页增加无实体数据时的空态渲染
- 确保按钮仍可进入页面
- 确保无解释诗篇不报错、不空白

### 第三阶段：文本精修标准落地

- 按新的说明文标准重写生成模板
- 让 `summary / facts / details` 都更偏向对象本身
- 对已有人工精修条目应用同一标准，避免风格断裂

### 第四阶段：回归校验

- 抽查高频诗篇，确认高亮对象没有误删
- 抽查空态诗篇，确认解释页行为正常
- 抽查动植物条目，确认图片仍可用
- 抽查人物、地名条目，确认文本显著增厚

## 风险与处理

### 风险 1：过滤后部分诗篇解释页过空

处理：

- 接受“没有这四类对象”的事实
- 通过空态页保持入口连续性
- 不为了避免空态而硬塞不合适对象

### 风险 2：旧手工数据与新生成数据风格不一致

处理：

- 在合并层统一过滤类别
- 对重点手工条目按新标准补写
- 允许短期内存在少量密度差异，但语气必须统一

### 风险 3：解释文字变长后阅读压力上升

处理：

- 保持段落短而清楚
- 不做百科式大段术语堆积
- 用自然说明文控制密度，而不是靠标题分节

## 验收标准

- 解释系统中不再出现 `器物 / 礼制 / 制度 / 天象`
- 全书解释页仅保留 `人物 / 地名 / 植物 / 动物`
- 无相关解释对象的诗篇进入解释页后显示统一空态说明
- 动植物图片链路不回退
- 新生成解释的主体内容明显更详尽，并以对象本身的说明为主
- `npx tsc --noEmit` 通过
- `npm run build` 通过
